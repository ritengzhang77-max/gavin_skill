# Research Results Synthesis and Explanation

## Public showcase attribution

This is a Gavin/Hermes local workflow skill, substantially shaped by Gavin-specific research projects, artifact conventions, and repeated claim-hygiene corrections. It is not presented here as a stock upstream Hermes package skill.


Use this skill when the user asks to explain, interpret, summarize, or review results from a research project, especially when the results are stored across local notes, CSVs, markdown reports, plots, generated decks, or run directories.

This is for **research interpretation**, not just file summarization.

## Project-specific references

- For Value Action / Phase 3.1 ConsVal artifact curation and tier explanations, see `references/value-action-consval-artifact-scope-and-tiering.md`. Use it to avoid conflating top-32 triage artifacts with full-support dense artifacts, to explain lenient/default/strict percentile bars, and to sanity-check order-direction/order-robust labels before answering.
- For nonlinear SAE paper-readiness / old-theory questions about false positives, false negatives, support-selection errors, angular distance, mean-direction divergence, parameter-matched wider-linear 60-row maps, or whether ResNet solves a linear-encoder pathology, see `references/nonlinear-sae-support-selection-pathology-metrics.md`. Lead with the exact readiness/CUDA answer, then frame the claim as “nonlinear encoders improve sparse support selection under strict budgets,” not “nonlinear SAEs are universally better.”

For terse Discord project status follow-ups (e.g. “again”, “is it still running?”, “check the jobs”), use the fast GPU/process/log/progress-file recipe in `references/quick-gpu-status-checks.md`: infer the project from the channel/router, map GPUs→PIDs→logs via `nvidia-smi`, `ps`, and `/proc/<pid>/fd`, verify liveness with log mtime/progress JSON rather than a single utilization sample, then answer with a compact verdict and small aligned fenced-text table.

## Interrupted Long-Run / Partial-Result Recovery Pattern

When a long research evaluation/training process is reported as completed with a signal exit (for example exit 143/SIGTERM) or otherwise stopped mid-run, do not assume failure or restart the full job. Use a recovery-first workflow:

1. Inspect the run log tail and searchable milestones to identify whether it stopped at a Python exception versus external termination/interruption.
2. Count completed output artifacts by eval/task family and compare them against the manifest rows or planned workload.
3. Identify the first missing row(s) and whether partial eval families lag behind others (e.g. one eval has 58 outputs while two have 60).
4. Create small resume manifests for exactly the missing prefix/next tranche instead of rerunning already-completed rows.
5. Relaunch a bounded continuation on an idle/appropriate GPU, reusing the same output root when the runner skips existing artifacts by default.
6. Verify the new process is actually using the intended GPU and record the process id/log path in the user-facing status.

Pitfall: for multi-eval manifests, interruption can leave eval families at different offsets. Resume the lagging eval(s) for the last partially completed rows separately before continuing the full eval set.

## Behavior / Format QA Gates Before Activation Collection

When synthesizing mechanistic-interpretability dataset QA that decides whether to collect answer-token/answer-marker activations, treat the behavior-only format gate as a hard prerequisite, not a side diagnostic. Verify the scored-row count, category rates, grouped breakdowns, and representative generations before recommending activation collection. If a large fraction of rows fail to start with the intended marker, explicitly say not to run answer-marker activations on that render and identify the best surviving prompt-format lane plus remaining failing controls. A good compact readout is: total rows scored, first-token-valid rate, critical bad-first-token rate, failure categories, artifact paths, best/worst format lanes, and the next repair. See `references/answer_format_qa_gates.md` for a reusable checklist and Value Action example.

 The goal is to reconstruct: what was tested, what changed, what the numbers mean, what is causally supported, what is merely correlational, and what claim is now safe to make.

## Workflow

1. **Find the right project and artifact root.**
   - If the user did not name a project, infer from the Discord channel, current working directory, project mapping notes, recent chat summaries, and modified/latest artifacts.
   - Prefer project-local `RECENT_CHAT.md`, `README.md`, `summary/`, `analysis/`, `results/`, `reports/`, and presentation folders.
   - For Gavin's research projects, project mapping notes may live under `/home/<user>/work/hermes-agent-work/`.

2. **Read synthesis docs before raw tables.**
   - Start with recent-chat notes, final synthesis markdown, analysis reports, plans, manifests, and deck READMEs.
   - Then inspect raw CSV/JSON tables to verify counts, means, top rows, and edge cases.
   - Do not rely on memory for counts or arithmetic; compute or read them from artifacts.

3. **For live experiment status, separate running state from completed results.**
   - Check live processes/GPU/tmux first, then progress files for durable done/failed counts, then logs for the current in-flight run, then artifact mtimes.
   - Also check user systemd services when runners were deliberately launched outside Hermes background processes, e.g. `systemctl --user list-units 'sae-*' --all`, and map PIDs back to the project root.
   - Use runner docstrings/configs to recover intended sweep size and ordering.
   - Treat `status: "running"` in a progress file as only a durable hint until corroborated by a live process/GPU/log; stale running entries should be reported as stopped/stale, not active.
   - For detached/systemd/nohup launches, a stale stdout log alone is not enough to call the job dead; map the live PID to its command/project root, check GPU utilization, and inspect output/checkpoint mtimes.
   - Ignore buffer-refill progress bars when estimating training progress; use the latest true training tqdm line and `progress.json` elapsed times.
  - For quick `x/100` progress checks, reconstruct the denominator from the runner's plan/config rather than `len(progress.json)`, because progress files may only contain touched runs; report `done_count + current_run_fraction` only when the running entry is live-corroborated, and also name the integer run index (`run n/100, p%`). Start with a compact count table before narrative: `done / running / failed / not-started`, plus the current run ID. If multiple sweeps are active, report per-sweep counts first, then a combined total only after the denominators are explicit.
  - When a user asks what `failed` means in a progress file, answer concretely: the runner caught an exception, wrote `status: "failed"`, skipped/continued to later rows, and usually did not create checkpoint/metrics artifacts unless verified otherwise. Immediately list the exact failed run IDs, their plan indices (`[97/100]` etc.), architecture/setting/sparsity values, first-line error, elapsed time if recorded, and whether expected artifacts such as `sae_weights.safetensors`, `cfg.json`, `final_metrics.json`, and `meta.json` exist. Then state runner behavior from code: e.g. if the loop skips only `done` rows, a retry runner will reattempt failed rows after reaching them.
  - When many live SAE/ML runs suddenly fail with generic CUDA errors, distinguish a model/config failure from a GPU/NVML/driver failure before retrying. If `nvidia-smi` reports an unknown GPU handle or fresh Torch processes cannot initialize CUDA even on a different GPU, stop retries, mark stale `running` entries honestly, preserve unaffected existing CUDA jobs, and recommend retrying failed rows only after a GPU/driver reset; see `references/live-sae-cuda-nvml-failure-triage.md`.
   - See `references/live-experiment-status-checks.md` for a reusable read-only workflow and answer shape, `references/live-experiment-stale-running-progress.md` for handling stale/detached `running` progress entries and incomplete progress-file denominators, and `references/nonlinear-sae-auxk-live-status-20260529.md` for the nonlinear SAE AuxK systemd-service / fractional `x/100` example.

4. **For benchmark/evaluation pipeline setup, treat launch readiness as part of synthesis.**
   - Use this same skill when Gavin asks to adapt an external benchmark, evaluate many model/SAE/checkpoint variants, prepare a GPU/API sweep, inventory metrics, estimate cost/runtime, or aggregate benchmark JSON into matched comparisons.
   - Resolve the authoritative benchmark repo/version; inspect source schemas/configs rather than README summaries; record branch/commit and compatibility caveats.
   - Inventory eval families and metric names before launching. Separate included, excluded, optional-but-needs-smoke, and incompatible metrics; answer “is X included?” with yes/no first.
   - Build manifest-driven runs with one row per checkpoint/run and explicit pairing metadata (model, layer/hook, architecture, baseline family, width, sparsity/k, checkpoint/config/output paths, human labels). Avoid ad-hoc shell loops.
   - Add adapter layers instead of mutating original artifacts; smoke both representative baseline and new-architecture checkpoints before full launches.
   - Treat LLM-judge/API evals as separate batches with cost/rate-limit pilots; do not bundle them with ordinary GPU metrics unless requested.
   - Shard with runner/manifest shard flags, use independent logs, and launch long bounded runs as tracked background processes with completion notification.
   - Aggregate pairwise before claiming wins; compare matched pairs first and do not collapse metric families into a single score until directions/relevance are explicit.
   - See `references/research-benchmark-pipelines.md`, `references/saebench-current240.md`, and `references/cuda-sweep-progress-eta.md` for the absorbed benchmark-pipeline workflow and concrete SAEBench/CUDA ETA examples.

5. **Reconstruct definitions and sign conventions.**
   - Explicitly state what each option/site/token/metric means.
   - For causal metrics, state the objective and sign convention before interpreting numbers.
   - Separate target labels, model choices, generated answer formats, action spans, rationale text, and intervention sites.

4. **Compute concise verification statistics.**
   - Use a small script to calculate counts, percentages, means/medians, positive/negative rates, and top/bottom examples from CSV/JSON artifacts.
   - If pandas is unavailable, use Python stdlib `csv`, `json`, `statistics`, and `collections` rather than stopping.

5. **Distinguish evidence classes.**
   - Separate raw activation patterns, validated feature explanations, global correlational behavior, case-study examples, and causal/intervention evidence.
   - Mark which results are strong, suggestive, mixed, or likely confounded.

6. **Explain as a research narrative.**
   - Start with a bottom line.
   - Then cover: scope, definitions, main quantitative results, qualitative examples, causal evidence, limitations, and the cleanest claim.
   - Include enough detail for the user to trust the interpretation, but avoid dumping every table.

7. **End with claim hygiene.**
   - Provide a conservative claim the user can make.
   - Identify what would be overclaiming.
   - Name the next validation experiment only if useful.

## Pitfalls

- **For Discord status/result tables, avoid Markdown pipe tables.** Gavin finds `|` table syntax hard to read in Discord. Render comparison tables as fenced `text` code blocks with aligned spaces. Keep this formatting especially for counts, run summaries, and metric comparisons.
- **Keep match units explicit.** When comparing experiment variants, state the key being matched and whether rows are run-level or collapsed. Example: AuxK SAE comparisons may have `width + L1 + aux coefficient` rows, while no-Aux standard-vs-linear uses only `width + L1`; do not compare denominators without explaining the extra axis.
- **For Gavin's SAE beat/loss questions, use Pareto beat/loss and split tradeoffs by direction.** A Pareto beat means the candidate is non-worse on both matched metrics of interest and strictly better on at least one (typically higher/equal EV and lower/equal L0, with at least one strict improvement). A tie on one metric plus a win on the other is a beat, not a tie. A Pareto loss is the analogous non-better-on-both with at least one strict worsening. Tradeoffs only mean strict EV-vs-L0 disagreement; report the direction explicitly, e.g. `candidate EV up / L0 worse` versus `candidate EV down / L0 better`, instead of collapsing all tradeoffs into one unlabeled count. If L0/sparsity is tied, EV decides the Pareto outcome: tied L0 + EV win is a beat; tied L0 + EV loss is a loss, not a sparse-only trade. Consider adding `L0 ties` and `L0 nonworse` columns when Gavin asks whether tied sparsity is being handled correctly. For the OPT-125M non-Aux core5 TopK/BatchTopK comparison, tied L0 in all 20 rows makes wins/losses pure EV outcomes; see `references/nonlinear-sae-core5-resnet-topk-batchtopk-design.md`.
- **For nonlinear SAE percentile-normalized comparisons, compute percentiles in fair pools, not inside pairs.** When Gavin asks for EV/L0 percentiles as a fair comparison layer, define a pool such as `family_group × width`, compute `EV_pct`, reversed `Sparse_pct` from L0, and a harmonic balanced score `H2 = 2*EV_pct*Sparse_pct/(EV_pct+Sparse_pct)`. For matched ResNet-vs-Linear comparisons, compare `ΔH2 = H2_candidate - H2_linear_baseline` after matching on width + sparsity setting (+ AuxK axis where relevant). Do not compute percentiles within each 2-row pair; that degenerates to 0/100 and adds little. Percentile transforms are monotone, so clean percentile beat/loss aligns with Pareto beat/loss; the new value is ranking tradeoffs and selecting balanced EV/L0 knee checkpoints. Include exact formulas and export summary + pair-row CSVs when making slides. See `references/nonlinear-sae-percentile-matched-comparison.md`.
- **For nonlinear SAE architecture comparison decks, keep candidate/baseline orientation stable.** When Gavin is comparing nonlinear/ResNet variants against matched linear baselines, label rows and slides as `ResNet vs Linear`, not `Linear vs ResNet`; compute deltas as candidate minus baseline (`ΔEV = ResNet EV - Linear EV`, `L0 improvement = Linear L0 - ResNet L0`). When both AuxK and no-Aux variants are shown, put AuxK rows first, then no-Aux. If asked for top runs across all architecture/settings by EV and L0, use a transparent rank-sum shortlist (`EV rank descending + L0 rank ascending`) and state that it is not a Pareto frontier proof. For PPTX layout details and the config sanity-check pattern for `apply_b_dec_to_input`, see the presentation skill reference `gavin-research-presentations/references/nonlinear-sae-resnet-vs-linear-pptx-comparison-decks.md`.
- **For nonlinear SAE fair metric-system questions, layer percentile normalization after matching and Pareto beat/loss.** Percentile scores are useful for selecting balanced EV/L0 knee checkpoints, but they are not standalone win/loss evidence. First match by fair keys and compute Pareto beat/loss; then compute EV and reversed-L0 percentiles inside local pools such as `family_group × width`, combine with a harmonic mean, and report dead-feature percentile as a separate sanity gate. Keep all-run top lists separate from nonlinear-candidate top lists. For `top N across all runs/architectures`, do not rank local family×width percentile scores as if they were absolute utilities: local percentiles can violate obvious cross-family Pareto dominance (higher EV and lower L0 can rank lower than a worse raw point from another pool). Instead compute a global EV/L0 percentile score over the declared candidate universe after applying any dead-feature eligibility filter, or clearly label the local-pool score as only a within-family knee selector. Dead neurons should be a filter/sanity gate, not part of the EV/L0 ranking score, unless Gavin explicitly asks for a dead-inclusive score. For the OPT Phase 1 result and GPT-2 Phase 2 replication plan, see `references/nonlinear-sae-global-ev-l0-ranking-and-phase2.md`; for parameter-matched wider-linear and pure nonlinear/no-skip control planning, see `references/nonlinear-sae-parameter-matched-baselines.md`; for the older local-pool percentile layer, see `references/nonlinear-sae-percentile-comparison-layer.md`.
- **For nonlinear SAE architecture-design questions, inspect implementation before inferring intent from names.** Matryoshka ResNet TopK in the current OPT-125M core5 code is a shared ResNet encoder with prefix reconstruction losses, not separate nested sub-ResNets; default levels are `[d_sae//4, d_sae//2, d_sae]`. Current/old `nonlinear_resnet_gated` uses an asymmetric design: ResNet gate path plus linear magnitude/value path. For linear gated specifically, explain that the gate and magnitude are parallel from `sae_in` but share `W_enc`; it is not gate-on-main-output and not two independent encoder matrices, so the parameter delta is only vector-sized (about `+2*d_sae`). `nonlinear_resnet_dual_gated` is a valid high-capacity ablation with independent ResNet gate and magnitude/value paths, but it is **not** the closest direct analogue of ordinary linear gated. If Gavin asks for the “actual corrected” or “core/final” ResNet-gated design, propose/label a tied/shared ResNet-gated architecture: one shared ResNet encoder output `z`, featurewise `gate_pre = z + b_gate`, featurewise `mag_pre = z * exp(r_mag) + b_mag`, and `acts = (gate_pre > 0) * relu(mag_pre)`. Do not collapse this to `(z > 0) * relu(z)`, which degenerates to standard ReLU ResNet. Treat dual-gated as a separate dual-path ablation, not the canonical fixed gated result unless Gavin explicitly decides otherwise. For parameter-matched wider-linear or makeup-run planning, combine trainable-param counts with a rough CUDA fit screen and verify fresh PyTorch CUDA init before launching if NVML/GPU health is suspicious. When adding new custom SAE architectures for these sweeps, verify both the training registry and the inference/base SAE registry before launch; dry-run config validation alone may miss final-save `KeyError` failures. See `references/nonlinear-sae-core5-resnet-topk-batchtopk-design.md`, `references/nonlinear-sae-gated-design-and-cuda-fit-check.md`, `references/nonlinear-sae-tied-resnet-gated-and-reboot-run-plan.md`, `references/nonlinear-sae-tied-gated-design-and-runner.md`, and `references/nonlinear-sae-custom-architecture-runner-pitfalls.md`.
- **For nonlinear SAE live runner launches, keep status compact and make runs resumable.** Before starting additional long sweeps, check live GPU/process state and progress files, avoid historically unstable CUDA2 unless explicitly needed, and use one physical GPU per runner with `CUDA_VISIBLE_DEVICES=<physical_gpu>` plus `--device cuda:0`. Use conservative thread caps (`TORCHINDUCTOR_COMPILE_THREADS=4`, `MAX_JOBS=4`, `OMP_NUM_THREADS=4`) for multi-day jobs. If a runner dies or is killed, back up `progress.json`, mark stale `running` rows as failed only after live-process corroboration fails, and relaunch; these runners should skip `done` rows and retry failed rows. Report `done/running/failed/not-started` and current run ID first. See `references/nonlinear-sae-custom-architecture-runner-pitfalls.md`.
- **When AuxK is involved, compare each method pair separately and label scope.** For early standard-family AuxK pilot results, produce separate rows/plots for `AuxK ResNet vs linear`, `AuxK ResNet vs no-Aux ResNet`, and, if relevant, `no-Aux ResNet vs linear`, with pilot denominators clearly labeled. For expanded all-core5 AuxK sweeps, recompute exact all-core5 matches (`family + width + sparsity parameter/value`, with the AuxK coefficient fixed/explicit) and update all denominators from live artifacts rather than reusing old pilot counts. If only one side of a fourth comparison is partial, e.g. `ResNet AuxK vs Linear AuxK`, report that pair with its current matched denominator separately and do not fold it into the completed three-way PNG/table.
- **Do not collapse sites.** In interpretability results, action spans, pre-answer tokens, generated answer tokens, and generated rationales can mean different things. Keep them separate unless the artifact explicitly combines them.
- **Do not treat generated rationales as the decision mechanism.** They can be post-answer traces and should be interpreted separately.
- **Do not interpret raw activation rank as semantic importance.** Generic syntax, pronoun, formatting, and scaffold features can dominate raw top-k lists.
- **Do not ignore answer-format confounds.** If a run varies the answer format, compare choice distributions and answer-site feature stability before interpreting answer-token features.
- **For final live-status checks, verify artifacts, not just progress counts.** When a sweep transitions from active to complete, report live process/service state first, then compare `progress.json` done counts with expected `final_metrics.json` artifacts. A run can be marked `done` and have weights/meta/log completion but lack `final_metrics.json`; report that as a metrics-file recovery caveat rather than a training failure if logs and checkpoint files corroborate completion. If Gavin says a previously-live GPT/nonlinear-SAE tail is now finished and asks to move into analysis, re-verify the final state first, then regenerate summaries/decks from final metrics rather than relying on provisional artifacts; see `references/nonlinear-sae-final-status-before-analysis.md`. For nonlinear SAE makeup-sweep completion and CUDA visibility recheck details, see `references/nonlinear-sae-live-multisweep-status-20260604.md`. For GPT-2 pure-nonlinear no-Aux final completion after a registry/final-save fix, verify both `progress.json` and all 100 checkpoint artifact dirs across `2layer_4k` and `2layer_8k` before moving to analysis; see `references/nonlinear-sae-gpt2-pure-nonlinear-final-completion-20260630.md`.
- **For nonlinear SAE dominance/DAME/redundancy follow-ups, start with fixed-`k` TopK-family rows and require ranking sanity gates before expensive downstream metrics.** Gavin prefers the first dominance-aware interpretability pass to focus on BatchTopK, corrected Matryoshka TopK, and original TopK because fixed `k` makes ResNet-vs-Linear wins/losses much cleaner: sparsity is controlled and EV decides the row. Treat standard/gated L1-family rows as secondary/appendix unless a separate L0-binned/thresholded comparison rule is defined. Use a manifest-first sequence, but do **not** proceed from manifest to DAME/redundancy until dominance ranking passes basic invariants: eval L0/k matches `final_metrics.json`, active feature counts are plausible, sample size is sufficient for rank/quantile splits, and split-half rank/tertile stability is reported. When dead neurons or redundancy are used as “not too bad” filters in ResNet-vs-Linear tables, distinguish the side with the problem: mark a true candidate failure only if the ResNet/candidate side violates the threshold; Linear-only high-redundancy rows should be labeled as baseline issues and still compared on EV/DAME. Include standalone redundancy/dead-neuron count summaries before interpreting whether redundancy is getting better or worse. The failed v1 ranking used only 10 batches / 5,120 positions and had thresholded OPT BatchTopK/Matryoshka rows with high EV/L0 encode as all-zero/near-zero; downstream DAME and redundancy from those ranks were quarantined. Replacement should be named `Dominance V2 / sanity-checked dominance ranking`, use larger samples (about 50–100 batches for claim-facing ranks), checkpoint-type-aware encoding for thresholded BatchTopK/Matryoshka, and separate v2 output directories. Current DAME selection may use tertiles rather than quartiles when Gavin chooses presentation simplicity; build tertiles over active features only, but if active features are implausibly zero relative to final L0, mark the row invalid and rerun/fix the encoder rather than treating it as science. DAME score should be explained as token/sentence pairwise ranking accuracy from generated explanations, not percent of features correct. For “top 10 overall/Q1” requests, rank all runs ignoring architecture and count ResNet vs Linear unless Gavin explicitly asks for ResNet-minus-Linear advantages. For improved claim-facing DAME reruns after V2, do not rerun dominance ranking if Dominance V2 already passed sanity; clarify that DAME “collection” means gathering activation examples for explanation/eval, not ranking. Gavin's current preferred V3 design is High-dominance top50 as primary, Mid-dominance top25 as robustness, Low omitted unless explicitly requested, with 100 collection batches/checkpoint, 30 explanation examples, and up to 40 token + 40 sentence MC questions; see `references/nonlinear-sae-dame-v3-high50-mid25.md`. For redundancy metric choice, prefer high-dominance Top100 decoder-cosine redundancy @0.5 as the headline count/rate, plus nearest-neighbor duplicate fraction and tertile-wide high/mid/low rates as robustness; do not use raw top100 pair count alone. When Gavin asks for EV+DAME comparison with dead-neuron/redundancy included only as “not too bad” filters, treat dead/redundancy as pair quality gates rather than ranking scores, and show separate green/red beat/loss columns for EV and each DAME metric instead of collapsing the row into one verdict; see `references/nonlinear-sae-ev-dame-quality-filter.md`. For dominance-level distribution / top-10 or top-30 cumulative dominance questions, answer the definition first: DAME-paper dominance is mean per-input activation share, `dominance_i = E_x[|z_i(x)| / sum_j |z_j(x)|]`, so top-N accumulation sums normalized shares after ranking features by this value. Do not describe it as raw accumulated activation magnitude or firing frequency. Also distinguish pure cumulative-dominance slides from redundancy slides that use dominance-ranked Top100 features but plot decoder duplicate counts. See `references/nonlinear-sae-dominance-level-distribution-v2.md`. For the full pattern and row/table presentation preference, see `references/nonlinear-sae-dominance-aware-topk-metrics.md`; for the 2026 tertiles/top25 DAME runner edge cases and GPT-2 interim analysis lessons, see `references/nonlinear-sae-dame-tertiles-run-and-analysis-202606.md`; for the v1 quarantine and Dominance V2 rerun standard, see `references/nonlinear-sae-dominance-v2-quarantine-20260627.md`. If Gavin asks whether to run DAME/redundancy on weak nonlinear ablations versus making ResNet DAME more robust, skip expensive interpretability metrics for variants that are not competitive on matched EV/L0, and for ResNet use a targeted DAME V3 robustness plan rather than a blind full rerun: increase high-dominance feature coverage, examples/questions per feature, and sampling/CI stability checks; keep Low-tertile claims caveated because V2 low-dominance coverage was poor. See `references/nonlinear-sae-dame-v3-robustness-plan.md`.
- **Do not overstate small-bundle interventions.** Exact Shapley over a selected bundle is exact for that bundle, not for the full feature basis.
- **For Value Action public SAE/transcoder/crosscoder basis-search status, separate scientific results from job state and separate the existing benchmark from newly tried pairs.** This is an RQ0-style preservation screen, not the full Keishii S/V/V3/V4 pipeline. Answer first whether a candidate is `BENCHMARK/current`, `STRICT`, `STRICT_DENSE`, `RELAXED`, `running`, `FAIL`, or `BLOCKED before science`; do not list pending or blocked candidates as winners. When Keishii L11 appears in the table, label it as `BENCHMARK/current pair`, not as a newly discovered candidate; count totals as `benchmark/current`, `new/other screened`, `new strict passes`, `strict-dense numeric passes`, `relaxed candidates`, and `fails` so Gavin does not think the current pair is being rediscovered. For near/relaxed rows, always show the exact percentage preservation plus the two corresponding scores that produced it, e.g. `raw acc 0.781 → sparse acc 0.656 = 84.0%`, and do the same for macroF1 and matched-N when available. Use aligned text-table columns: LLM name/size, basis repo/type/width, layer site, checkpoint size, raw acc/macroF1, sparse acc/macroF1, sparse/raw preservation, raw top-N MI, sparse top-N MI, matched-N preservation, L0/active features, and decision. Keep two bars distinct: strict/gold promotion generally needs sparse acc or macroF1 `>=0.70` and `>=80–85%` sparse/raw preservation; Gavin may also ask for relaxed exploration around sparse acc/F1 `>=0.60` and preservation `>=80%` (GemmaScope L12-style candidates). A roughly 30-row RQ0 public-basis screen is enough for basis-selection evidence, but not final causal/steering claims; after such a screen, promote only representative passing settings rather than every correlated L0 checkpoint. If a candidate passes strict numeric thresholds but has very high observed L0/active feature count, label it `STRICT_DENSE` and caveat that it may be less interpretable / less cleanly sparse than Keishii. If a candidate cannot run due gated model/auth, label it `BLOCKED before scientific result`, not a scientific failure. See `references/value-action-public-basis-result-table.md`, and `references/value-action-public-basis-30-screen-20260626.md`. 
- **For Value Action dataset redesign / value-action-gap dataset validity, require genuine two-sided value conflict and baseline specificity before strong claims.** If Gavin raises concerns that current Task1/Task2 items are not controversial enough, answer that the existing dataset should be treated as a pilot/warning signal: weak Task1↔Task2 overlap may reflect prompt format, option position, context artifacts, generic moral-question features, or weak value-action mapping rather than a real value-action gap. The preferred new unit is a paired controversial value conflict: Task1 asks abstract priority between two defensible values, and Task2 presents a concrete dilemma where each action instantiates one value.
