# Thread Canvas / Discord Artifact Workflow

Gavin's Discord workflow treats each research/project thread as a visible control surface.

## Human-facing artifact policy

- Short answers go directly in Discord.
- Long research/project answers should be backed by a Thread Canvas / web-log page when the channel supports it.
- Files panel should contain only the human-facing artifact set: deck, report, dashboard, key figures, not every intermediate scratch file.
- Normal long text should not be sent as random `hermes-response-*.md` attachments.

## Visual expectations

- User and assistant messages in clean-chat logs should be visually distinguishable.
- Thread Canvas file names should keep real extensions (`.html`, `.pdf`, `.png`) so they preview/open correctly.
- Mobile-friendly clean aliases are preferred when long generated URLs are awkward.

## Typical artifact lifecycle

1. Produce answer/deck/report locally.
2. Copy exportable artifacts to an exports directory.
3. Register or link the polished files in the thread canvas / web log.
4. Verify links render as `text/html` or native media where expected.
5. Summarize in Discord with direct links and concise bullets.

## Privacy note

Thread Canvas pages are excellent for workflow demonstration, but real thread logs can contain private project details. Public examples should be synthetic or manually redacted.
